## 1. People & Roles

> Who are in the team? What are their roles?

- **넝녕 (이은영 → 영영 → 넝녕)** — 프론트엔드 개발 재직 만 2년. 법학 전공 후 개발로 전향. 남동생 2명, 취미는 보리수 먹기·피아노. "이번 스프린트는 재밌게·재밌게·재밌게 만들고, 서로 AI 활용법 공유하고 개발 경험 듣고 싶어요."
- **류하** — 산업공학 전공. 레고 만들기 좋아함. "AI 활용해서 재밌게 작업하면 좋겠어요~!"
- **메굼 (메굼밍밍)** — 커뮤니케이션 디자인학과, 디자이너 → 개발자. 기획·디자인·프론트 개발까지 커버. 팀 분위기 메이커(본인 피셜 광대). "개발자가 AI에 대체되면 직업 뭐하지?"를 고민.
- **MC(진행자)** — 팀 MC를 정하는 중. *유의사항: MC는 진행하는 사람이지 **결정**하는 사람이 아니다 / 누군가의 말을 **요약**하기보다 모두가 좋은 의견을 낼 수 있게 이끌어준다.*

## 2. Team Goals

> What do we want to achieve? How do we measure success?

- **다음 주까지 서비스 출시(배포)하기** — 3명이 일주일 안에 배포까지 가는 게 목표.
- **싸우지 않고 즐겁게** — 웃으면서 작업하기 / 얼굴 붉힐 일 없이 쿠션어 쓰기 / 그래도 완성은 보기.
- **협업에서 인사이트 얻어가기** — 서로 추구미 보드 만들어주기 포함.
- 성공 지표(측정): 실제 **배포 완료** + 완성된 결과물, 그리고 "낼도 보고 다음 주에도 얼굴 보는" 관계. *(TODO: 정량 지표는 미정)*

## 3. Personal Goals

> What personal goals do we have?

- **넝녕** — 회사에서 쓰는 vue 말고 다른 스택 써보기 / 기획·디자인·개발 AI flow 문서 만들기.
- **류하** — AI 활용해서 기획부터 개발까지 더 빠르게 할 수 있으면!
- **메굼** — 기획 플로우 정립 → AI 도구로 만들어두기 / 프론트엔드 협업용 AI 작업 환경 구축 / 기획→개발(백·프론트)→배포까지 **AX(개발 경험)** 설계 많이 생각해가기.

## 4. Purpose

> Why was this team created? Why is this worth working on?

- **한 줄로: "공유, 내 무드"** — 내 취향(무드)을 시각화하고 친구들끼리 공유할 수 있는 서비스를 만들기 위해.
- 추구미를 형(현)영으로 표현하고, 우리끼리 공유하기 편한 걸 만들고 싶다 → 나의 색깔을 알고, 서로에게 추구미를 공유하는 서비스로.
- 주변 사람들과 소통하고 **나를 더 알기** 위해.

## 5. Values

> What values do we share? How do we want to work together?

- **결국 이뻐야 해요!!** — 감성·"느좋(느낌 좋은)" 결과물이 중요. 핀터레스트·외국 감성 지향.
- **쉬운 진입 → 낮은 이탈** — ① 공유된 무드보드를 보고 나도 써보고 싶게 ② 진입한 사람이 바로 핵심 기능까지 도달(이탈율 낮게) ③ 초기 플로우는 복잡하지 않고 쉽게 ④ 개인 커스텀은 필요할지 논의 ⑤ 결국 이뻐야 함.
- **쉽게 접근·이용**할 수 있어야 하고, 결과물이 무엇보다 중요.
- (톤 논의: "밤티(반티) 안 됨.. 근데 밤티가 들 수도.. 밤티도 누군가의 추구미" — 유행 이미지에 매몰되지 않되 취향은 존중.)

## 6. Strength & Assets

> What abilities and skills do we have that can help us?

- **넝녕** — 결정을 잘함(장단점 캐치 후 빠른 선택, 효율 중시, 극J) / 계획을 잘 세움 / 정리정돈(회의·문서·코드) 잘함 / 잘 먹음(생오이·생굴 빼고 다).
- **류하** — 계획 세우기 / 화를 잘 안 내는 편.
- **메굼** — 분위기 메이커(본인 피셜 광대) / 회의 정리·공유 잘함 / 기획·디자인·프론트 개발까지 가능 / 개인 맞춤형 커뮤니케이션 / 진행 밀어붙이기(스케줄 병목 관리) / 유연함(의사결정도 코드도).
- 팀 공통: 커뮤니케이션은 다수보다 **1:1에서 더 강세**.

## 7. Weaknesses & Risks

> What are the weaknesses we have? What should our teammates know about us?

- **넝녕** — 말이 느린 편이라 답답할 수 있음 / 말보다 글 / 아침형이라 밤에는 취약 / **react·next.js 실무 사용은 안 해봤음**.
- **류하** — 결정을 잘 못함 / react를 실무 아닌 프로젝트로만 써봄.
- **메굼** — 귀가 잘 안 들림 / 감정이 격해지면 목소리가 커짐 / 꼼꼼히 해도 버그 있을 수 있음 / 생각·아이디어가 과하게 나올 수 있음 / 설명을 조리 있게 잘 못함.
- **기술 리스크**: 팀 다수가 **React 실무 경험이 적음** → React/Next.js를 이번에 새로 익혀야 함.

## 8. Needs & Expectations

> What do we need from each other? What do we expect?

- **"나 사용설명서"** — 각자의 특성을 미리 공유해 오해를 줄인다.
- **류하** — 결정장애가 있어 망설이면 "고민 중"으로 봐주기.
- **메굼** — 아이디어를 많이 던지는 편이니 적당히 걸러 들어주기 / 귀가 잘 안 들림 / 비유를 강하게 하는 편.
- **넝녕** — 뜸 들일 때가 있는데, 머릿속에서 로딩 도는 중이라고 생각해주기.
- 공통 기대: Weakness를 토대로 **미리 말하고**, 상대가 그걸 알고 배려해주는 팀.

## 9. Rules & Activities

> What rules do we have for communication? Rules for meetings? Any regular activities?

- **소통 규칙** — 코드 수정 시 말하기 필수!! / 간략한 설명 + 추가 질문에 따른 상황 공유 / 잡담과 집중할 시간을 구분해 미리 이야기하기.
- **분위기** — 대학교 동아리 야작 느낌으로 친밀·즐겁게, 그러나 결과물은 쌈뽕하게. 친밀한 팀 ㅎㅎ.
- **주말 에너지 투자 지수** — 류하 8점 / 메굼 8점 / 넝녕 8.2점 (주말에도 꽤 몰입 가능).
- **범위 합의** — 최소: MVP + 배포 + 스프린트 사용 / 최대: 기능 늘려 업데이트 출시, 서로의 추구미 보드 만들어주기.
- 어떻게 기억되고 싶은 팀? → "소수정예 협업 짱이었던 팀", "3명이서 일주일 만에 서비스 배포한 팀".
